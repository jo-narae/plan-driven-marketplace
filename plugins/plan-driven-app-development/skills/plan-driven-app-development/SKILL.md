# Plan-Driven App Development

<HARD-GATE>
## 절대 규칙 — 이것을 어기면 안 된다

1. **한 번에 하나의 Phase만 진행한다.** 여러 Phase를 한 응답에서 처리하지 않는다.
2. **각 Phase 시작 시 반드시 먼저 질문한다.** 질문하지 않고 문서를 쓰지 않는다.
3. **사용자 답변을 받기 전에는 문서를 작성하지 않는다.**
4. **문서 작성 후 사용자 확인을 받기 전에는 다음 Phase로 넘어가지 않는다.**
5. **Phase를 합치거나 건너뛰지 않는다.** "간단하니까 바로" 하지 않는다.

위반 시 사용자가 요구사항·페르소나·시나리오를 확인하지 못한 채 결과만 받게 된다.
이 스킬의 존재 이유가 사라진다.
</HARD-GATE>

## 앱 특성 기반 질문 생성 — 단조로움 방지

정형화된 항목(스택·저장 방식·디바이스·시간 예산)은 모든 앱에 공통이라 **고정 옵션**으로 유지한다.
그러나 **이 앱에서만 의미 있는 결정**(요구사항 본질·페르소나·시나리오·차별화 장치)은
앱마다 갈리므로, 매 Phase에서 **고정 질문 위에 앱 특성 질문을 얹어야 한다.**

같은 카테고리(예: "할일 앱")라도 결정 지점이 다르다 —
가계부면 카테고리 체계와 통계 단위, 갤러리면 메타데이터와 검색 축,
회의록 앱이면 입력 방식과 요약 자동화 — 같은 4가지 템플릿 질문만 던지면
사용자는 매번 비슷한 답을 하고, 결과물도 평균값에 수렴한다. 그게 이 스킬의 단조로움 문제의 근원이다.

### 질문 만들기 3단계

생각이 필요한 Phase(1·2·3·4)를 시작할 때마다 거친다:

1. **앱 컨셉 파악**: 사용자의 첫 발화/Phase 0 답변에서 도메인·핵심 가치·주 사용자를 추출한다. 모호하면 한 줄 정의부터 묻고 답을 받는다.
2. **이 앱 고유의 불확실성 식별**: 같은 도메인 안에서도 갈라질 수 있는 결정 지점 3~5개를 떠올린다. "이 답이 바뀌면 결과물이 크게 달라지는가?"가 기준.
3. **앱 특성 질문 작성**: 그 불확실성을 짚는 질문 3~5개를 만든다. 선택지 2~4개씩 제시하고 추천에는 "(추천)" 표시. 같은 4가지 질문 템플릿을 그대로 복붙하지 않는다.

`references/phases.md`의 예시는 출발점일 뿐, **이 앱에 맞게 다시 만든다.**

## 각 Phase의 실행 패턴

모든 Phase는 반드시 이 4단계를 거친다:

```
① 질문 (공통 + 앱 특성) → ② 답변 대기 (STOP) → ③ 문서 작성 → ④ 확인 요청 (STOP)
```

- ① 질문: 정형화된 공통 질문(고정 옵션) + 앱 특성 질문(3~5개, 위 "앱 특성 기반 질문 생성" 3단계로 생성)을 한 응답에 묶어서 묻는다. 각 옵션이 뭘 바꾸는지 설명하고 추천에는 "(추천)" 표시.
- ② STOP: 사용자가 답할 때까지 아무것도 하지 않는다.
- ③ 문서 작성: 답변을 반영해 해당 Phase의 `docs/` 파일을 작성한다.
- ④ STOP: "Phase N 완료. 핵심: [한 줄]. 다음 Phase로 넘어갈까요?" — 사용자가 OK할 때까지 멈춘다.

질문 형식 참고: `references/phases.md`

## Phase 0 — 착수 확인

사용자가 새 앱을 만들자고 하면 아래 네 가지를 묻는다 (앱 컨셉이 첫 발화에 충분히 들어있으면 1번은 인용 확인):

1. **앱 컨셉 한 줄**: "이 앱을 한 줄로 정의해주세요" — 도메인·핵심 가치·주 사용자가 이 한 줄에서 잡혀야 이후 Phase의 앱 특성 질문을 만들 수 있다.
2. **시간 예산**: 반나절? 하루? 며칠? (스코프 결정에 필요)
3. **문서 저장 위치**: 기본값 `docs/` — 다른 곳 원하는지?
4. **전체 흐름 안내**: "7단계(요구사항→페르소나→시나리오→차별화→디자인→아키텍처→계획)로 먼저 기획 문서를 만들고, 그 다음 마일스톤으로 나눠 구현할 거예요. 각 단계마다 질문드리고 확인받으면서 진행합니다."

## Phase 1 — 요구사항 (`01-requirements.md`)

질문은 **두 묶음**으로 만들어 한 응답에 함께 묻는다.

**A. 공통 질문 (고정 옵션, 모든 앱 공통)**:
- 이 앱의 주된 목적? (개인 / 포트폴리오 / 팀 내부 / 대외 배포)
- 주 사용 디바이스? (데스크톱 / 모바일 / 반응형)
- 기술 스택 선호? (React+TS / Next.js / Vanilla / 기타)
- 데이터 저장 방식? (localStorage / JSON / SQLite / 원격 DB)

**B. 앱 특성 질문 (3~5개, 이 앱에 맞춰 직접 생성)**:
"앱 특성 기반 질문 생성" 섹션의 3단계로 만든다. 같은 카테고리라도 도메인별로 결정 지점이 다르다 — 아래는 출발점 예시일 뿐, 이 앱에 맞춰 새로 만든다:
- 가계부 앱: 카테고리 체계의 자유도 / 입력 단위(거래·일·요약) / 통계 시각화 우선순위 / 카드·계좌 외부 연동 여부
- 포토 갤러리: 메타데이터 노출 범위(촬영일·위치·태그) / 정렬·필터 기준 축 / 일괄 작업 단위 / 클라우드 동기화 정책
- 회의록 정리: 입력 방식(타이핑·음성·붙여넣기) / 요약 자동화 정도 / 공유 단위(팀·개인) / 후속 액션 추출 여부
- 학습용 플래시카드: 복습 알고리즘(SM-2·Leitner·간단 반복) / 카드 형식(텍스트·이미지·오디오) / 진도 측정 기준 / 데크 공유

같은 4가지 템플릿 질문을 그대로 쓰지 말고, 앱 컨셉에 맞는 결정 지점을 골라낸다.

## 구현 단계

### M1 시작 전: plan mode

사용자에게 "Shift+Tab으로 plan mode 켜주세요" 안내.
plan mode에서 M1 실행 순서를 작성하고 승인 요청.

### M1~M(N-1): 마일스톤별 실행

각 마일스톤은 반드시 이 패턴을 따른다:

```
① 구현 → ② 자동 테스트 + 수동 확인 → ③ 결과 보고 (STOP) → ④ 사용자 OK 후 다음 M
```

- 구현이 끝나면 자동 테스트를 돌리고, 브라우저 확인이 필요하면 사용자에게 안내한다.
- **"M[N] 완료. [핵심 변경]. 브라우저에서 [확인 포인트] 확인해주세요. 다음 M[N+1]로 넘어갈까요?"**
- 사용자가 OK하거나 수정 요청할 때까지 다음 마일스톤을 시작하지 않는다.
- 수정 요청이 오면 해당 M 안에서 수정 후 다시 확인을 받는다.

### 최종 마일스톤: 다시 plan mode로 QA

QA 체크리스트:
- A. 자동 검증 (test / build / lint)
- B. 페르소나 시나리오 재현
- C. 모바일/반응형
- D. 엣지 케이스 (데이터 영속, 한글 IME, 빈 입력, 긴 입력)
- E. 마감 (README, .env.example, .gitignore)
