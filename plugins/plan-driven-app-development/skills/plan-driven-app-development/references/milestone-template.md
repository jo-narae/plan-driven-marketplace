# 마일스톤 문서 & plan mode 템플릿

## 마일스톤 개요 표 (07-plan.md)

```markdown
## 마일스톤 개요

| # | 목표 | 끝나면 보이는 것 | 예상 |
|---|---|---|---|
| M1 | 셋업 + 최소 CRUD | 기본 동작 화면 | 40~60분 |
| M2 | [도메인 필드 + UI] | 배지·카드 완성 | 40~60분 |
| M3 | [필터·정렬·그룹핑] | 목록 조작 가능 | 30~40분 |
| M4 | [고급 기능 X] | X 동작 | 20~30분 |
| M5 | [엔게이지먼트 요소] | 진척 가시화 | 15~20분 |
| M6 | 최종 점검 | QA 통과 | 15~20분 |

**합계**: 약 2~3시간
```

## 마일스톤 상세 템플릿

```markdown
## M[N] — [제목] ([예상 시간])

**목표**: [한 문장으로 상태 전환 기술]

**할 일**
- [ ] 파일 A 생성/수정
- [ ] 파일 B 생성/수정
- [ ] ...

**검증 방법**
- [ ] 수동: [브라우저에서 확인할 시나리오]
- [ ] 자동: [추가할 단위 테스트]

**중단 가능 지점**: 이 M 끝이 안전 중단 지점. 여기까지면 [어떤 수준]으로 쓸 수 있음.
```

## plan mode 전환 안내 문구

사용자에게 plan mode를 요청할 때:

> Shift+Tab을 눌러 plan mode로 전환한 뒤 "M1 시작하자" (또는 "최종 QA 시작하자")라고 말씀해주세요. 제가 코드를 쓰지 않고 먼저 계획만 제시한 뒤 승인받고 진행합니다.

## plan mode 내 플랜 파일 구성

plan mode에서 작성하는 플랜 파일(각 세션마다 자동 생성되는 `.claude/plans/*.md`):

```markdown
# M[N] — [제목] 구현 계획

## Context
- 왜 이 M을 하는지, 직전 M에서 완료한 것
- 기준 문서 위치 (06-architecture.md, 07-plan.md)
- 예상 시간
- 중단 가능 지점

## 범위 (이 M에서 하는 것)
- 구체 항목 N개

## 이 M에서 **하지 않는** 것
- 다음 M으로 미루는 것 (혼란 방지)

## 실행 순서
1. 환경 준비 (의존성 설치 등)
2. 폴더 스켈레톤
3. 핵심 파일 내용 (간략)
4. 테스트
5. 수동 검증

## Critical Files (생성/수정)
- 절대 경로 목록

## Verification (End-to-End)
1. `npm test`
2. `npm run build`
3. 브라우저 수동 체크 시나리오

## 의도적 설계 결정
- 어떤 선택을 했고 왜 (e.g. debounce 생략, 특정 패키지 채택 등)
```

## 최종 QA(M_N)용 체크리스트 템플릿

```markdown
# M[N] — 최종 QA 체크리스트

## Context
- 이 단계는 "기능이 다 됐다"를 넘어 사용 가능성을 확인하는 단계
- plan mode인 이유: 체크리스트 합의 자체가 산출물

## A. 자동 검증 (~3분)
- [ ] `npm test` 전부 통과
- [ ] `npm run build` 성공
- [ ] `npm run lint` 0 errors

## B. 페르소나 시나리오 재현 — 데스크톱 (~8분)
각 시나리오 S1~SN을 실제 브라우저에서 재현.
- [ ] S1 ...
- [ ] S2 ...

## C. 반응형 / 모바일 (~4분)
DevTools 모바일 뷰포트로 전환.
- [ ] 가로 스크롤 없음
- [ ] 터치 영역 ≥ 40px
- [ ] 주요 UI 깨짐 없음

## D. 엣지 케이스 (~5분)
- [ ] 데이터 영속 (새로고침 후 유지)
- [ ] 네트워크 실패 폴백
- [ ] 한글 IME 조합 중 Enter 이중 입력 방지 (한국어 앱)
- [ ] 빈 입력 무시
- [ ] 매우 긴 입력 처리

## E. 마감 작업 (~3분)
- [ ] README 프로젝트용으로 교체
- [ ] 환경 변수 예시 파일 (.env.example 등)
- [ ] .gitignore에 시크릿 확실히 포함
- [ ] 불필요한 console.log 정리
- [ ] git 상태 점검

## 버그 발견 시
- 경미(오타·색): 즉시 수정 후 재점검
- 기능 버그: "당장 고칠지 vs v2" 사용자와 합의
- 범위 외 개선: 메모만 남기고 v2로
```

## 마일스톤 종료 절차

각 M 끝에서:
1. 자동 테스트 → 녹색
2. 브라우저 수동 체크 → 녹색
3. (권장) git commit — 한 마일스톤 = 한 커밋. 메시지 첫줄: `M[N]: [제목]`
4. TaskCreate 태스크 `completed` 처리
5. 사용자에게 한 줄 보고: "M[N] 완료. [핵심 변경]. 다음은 M[N+1]"
