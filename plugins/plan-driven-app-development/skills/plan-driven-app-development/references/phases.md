# Phase 별 템플릿과 질문 예시

각 Phase를 시작할 때 참조한다. 아래 질문 예시는 **출발점일 뿐**, SKILL.md의 "앱 특성 기반 질문 생성" 3단계로 이 앱에 맞춰 재구성해서 사용한다. 같은 템플릿을 그대로 복붙하면 결과물이 평균값에 수렴한다.

## Phase 1: 요구사항

### 공통 질문 (모든 앱 공통, 고정 옵션)
- 이 앱의 주된 목적 (개인 사용 / 포트폴리오 / 팀 내부 / 대외 배포)
- 주 사용 디바이스 (데스크톱 / 모바일 / 반응형)
- 기술 스택 선호 (React+TS / Next.js / Vanilla / 기타)
- 데이터 저장 (localStorage / JSON / SQLite / 원격 DB)
- 시간 예산 (반나절 / 하루 / 여러 날)

### 앱 특성 질문 (3~5개, 도메인별 직접 생성)
같은 카테고리라도 결정 지점이 다르다. 도메인별 출발점:
- 가계부: 카테고리 자유도 / 입력 단위(거래·일·요약) / 통계 우선순위 / 카드·계좌 외부 연동
- 포토 갤러리: 메타데이터 노출 범위 / 정렬·필터 축 / 일괄 작업 단위 / 클라우드 동기화
- 회의록 정리: 입력 방식(타이핑·음성·붙여넣기) / 요약 자동화 정도 / 공유 단위 / 후속 액션 추출
- 학습 플래시카드: 복습 알고리즘 / 카드 형식 / 진도 측정 / 데크 공유
- 대시보드: 데이터 소스 / 새로고침 주기 / 주 KPI 3~5개 / 드릴다운 깊이

위 예시는 출발점. 사용자의 앱 컨셉을 보고 **그 도메인에서 결정이 갈리는 지점**을 새로 짚는다.

### 문서 섹션 표준
```
# 01. 요구사항 정의서
## 1. 프로젝트 개요 (표)
## 2. 기술 요구사항 (표)
## 3. 기능 요구사항
   3.1 데이터 모델 (표)
   3.2 핵심 CRUD
   3.3 ...
## 4. 비기능 요구사항
## 5. 범위 제외 (Out of Scope)
## 6. 완료 기준 (Definition of Done)
```

## Phase 2: 페르소나

### 질문 예시
- 주 사용자의 라이프스타일 (직장인 / 학생 / 프리랜서 / 일상)
- 사용 빈도 (하루 여러 번 / 한두 번 / 몰아서)
- 기존 도구의 가장 큰 불편함 (= 이 앱의 존재 이유)
- 대조 페르소나 타입 (다른 라이프스타일 + 다른 디바이스 비중)

### 문서 섹션 표준
```
# 02. 페르소나
## 주 페르소나: [이름]
   프로필 표, Goals, Frustrations, 행동 패턴
## 대조 페르소나: [이름]
   동일 구조
## 두 페르소나의 긴장 — 설계 기준 (축별 비교 표)
## 비-사용자 (타깃이 아닌 사용자)
```

## Phase 3: 시나리오

### 질문 예시
- 아침/오전/주요 사용 시점의 구체 행동
- "즉시 추가"하고 싶은 상황 (회의 중, 이동 중 등)
- 대조 페르소나에서 특별히 다른 사용 패턴

### 시나리오 작성 패턴
각 시나리오 1-2분 내 읽을 분량으로:
```
### S[N]. [짧은 제목] (페르소나)
> "[사용자 속마음 한 줄]"

**Context**: [어떤 환경·도구·시간]
**Flow**
1. ...
2. ...

**인사이트**: [설계에 어떤 요구를 주는가]
```

### 설계 원칙 집약
시나리오 끝에 **P1~PN 표**로 원칙을 모은다. 각 원칙은 한 줄 + 근거 시나리오 번호.

## Phase 4: 차별화·성취감

### 질문 (선택지 넉넉히)
"todo·다이어리·메모류 앱은 쓰다 말게 되는 경우가 많습니다. 어떤 장치가 있으면 계속 쓰게 될까요?" → 아이디어 10~14개 나열 후 3~5개 선택.

예시 카테고리:
- 성취감·보상 (진행률·스트릭·축하)
- 인지 부담 감소 (오늘 집중 핀·자동 정리)
- 입력 즐거움 (자연어·키보드 단축키)
- 변화·맥락 (시간대별 인사·빈 상태 축하)
- 부드러운 강제력 (시작 버튼·넛지)

### 문서 섹션 표준
```
# 04. 차별화·성취감 설계
## 한눈에 보기 (표)
## E1. [장치명]
   · 무엇을 / 어떻게 / 왜
## E2. ...
## 시나리오에 미치는 영향 (표)
## 도출된 추가 설계 원칙
## 의도적으로 **넣지 않는** 것
```

## Phase 5: 디자인 (3시안)

### 3시안 축 고르기
**레이아웃 구조와 분위기 둘 다** 시안마다 달라야 한다 — 한 축만 다르면 실패.
- 레이아웃 구조: 단일 컬럼 / 사이드바 / 카드 보드 / 그리드 / 분할 패널 등 **ASCII 와이어프레임 형태가 눈에 띄게 다른** 수준
- 분위기: 미니멀 / 명랑 / 진지 / 클래식 등 **타이포·여백·디테일 톤이 다른** 수준
- 안티패턴: 색 팔레트만 바꿔놓고 와이어프레임이 동일 → 사용자가 색 외 비교를 못 함 → 다시 생성
- 자기 점검: 세 ASCII 와이어프레임을 나란히 놓고 봤을 때 구조가 진짜 다른가?

### 각 시안 구성
- 한 문장 철학 ("X에 극단적으로 집중")
- 컬러 무드
- ASCII 와이어프레임 (데스크톱 기준, 60~80자 너비)
- 핵심 특징 3~5줄
- 모바일 대응 방향
- 강점/약점

### 비교 표
```
| | 시안 A | 시안 B | 시안 C |
|---|---|---|---|
| 분위기 | ... | ... | ... |
| 레이아웃 | ... | ... | ... |
| 주 타깃 | 페르소나 X | 양쪽 | 페르소나 Y |
| 한 문장 | ... | ... | ... |
```

### mockups/index.html
세 시안을 **세로로 나열**한 단일 HTML. 각 시안은 별도 `<section class="concept">` 내부에 실제 UI와 비슷한 정적 렌더링. CSS 변수로 색 토큰 관리. 모바일 반응형까지 간단히.

### 확정본 전환
사용자 선택 후 `05-design.md`를 덮어쓰기:
```
# 05. 화면 설계 — 확정본
> 선택된 시안: [X]
## IA (컴포넌트 트리)
## 컴포넌트 명세 (각 컴포넌트의 구성·인터랙션)
## 상태 & 인터랙션 규칙
## 반응형
## 접근성
```

## Phase 6: 아키텍처

### 필수 포함
- 기술 스택 표 (선택 + 이유 + **비채택 이유**)
- 폴더 구조 (트리, 파일명까지)
- 타입/스키마 (실제 코드 블록)
- 상태 흐름 다이어그램 (ASCII)
- 저장소 스키마 (localStorage 키, 버전 마이그레이션 등)
- 외부 API 연동 시:
  - 흐름 다이어그램
  - 프롬프트/요청 구조
  - 보안 고려 (키 노출 방지, rate limit)
  - 실패 폴백 전략

## Phase 7: 구현 계획

### 필수 포함
- 마일스톤 개요 표:
```
| # | 목표 | 끝나면 | 예상 |
|---|---|---|---|
| M1 | 셋업+CRUD | 기본 동작 | 1h |
| M2 | ... | ... | ... |
```

- 각 마일스톤 상세:
```
## M[N] — [제목] (예상 시간)

**할 일**
- ...

**검증 방법**
- [ ] 자동: ...
- [ ] 수동: ...

**중단 가능 지점**: (해당 시)
```

- **반나절 스코프에서 제외하는 것** 표 (v2 로드맵으로)
- plan mode 사용 규칙 (M1 시작 / 최종 QA)
